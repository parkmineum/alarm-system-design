# Claude Code 활용 흐름

이 프로젝트는 Claude Code 를 단순 자동완성 도구가 아니라 **워크플로우의 한 축** 으로 사용했다. 사람은 설계 의사결정과 톤 / 구성을 책임지고, Claude 는 구현·검증·셀프 리뷰·채점을 분담한다. 이 문서는 그 분담이 실제로 어떻게 굴러가는지 — 어떤 컨텍스트가 자동 로드되고, 어떤 슬래시 커맨드 / 서브에이전트가 정의되어 있고, 한 PR 이 어떤 단계를 거쳐 머지되는지를 정리한다.

---

## 1. 컨텍스트 로딩 — 자동 로드되는 파일들

세션이 시작될 때 Claude Code 가 자동으로 읽어들이는 파일은 두 종류다.

### 1.1 프로젝트 컨텍스트

| 파일 | 역할 | 핵심 효과 |
|---|---|---|
| [`CLAUDE.md`](../../CLAUDE.md) | 프로젝트 규약 / 코드 컨벤션 / 우선 참조 문서 | 모든 세션이 같은 규약에서 출발 |
| [`.claude/docs/assignment.md`](assignment.md) | 과제 원문 | 요구사항을 변형하지 않게 함 |
| [`.claude/docs/design-guideline.md`](design-guideline.md) | 설계 결정 사전 정리 | 구현 방향이 흔들리지 않게 |
| [`.claude/docs/roadmap.md`](roadmap.md) | 단계별 PR 계획 | 다음 작업 단위를 명시적으로 제시 |
| [`.claude/docs/git-conventions.md`](git-conventions.md) | 브랜치 / 커밋 / PR 규칙 | PR 작성 시 일관된 스타일 유지 |

CLAUDE.md 첫머리가 *"작업 전 반드시 두 문서를 먼저 읽어라"* 를 명시한다. 이 한 줄이 에이전트가 요구사항·설계를 무시하고 즉흥 구현하는 일을 차단한다 — 새 세션도 첫 발화에서 두 문서를 자발적으로 읽고 들어온다.

### 1.2 사용자 메모리 (`MEMORY.md`)

`~/.claude/projects/<프로젝트>/memory/` 에 영속되는 사용자 단위 메모리. 세션 간 학습이 살아남는다. 이 프로젝트의 메모리에는 두 항목이 있다.

| 메모리 | 학습 내용 |
|---|---|
| `feedback_no_coderabbit.md` | CodeRabbit 자동 리뷰 거부 — 새 PR / 저장소마다 `.coderabbit.yaml` 자동 추가 |
| `feedback_pr_style.md` | PR 작성 스타일 — 서술형 산문, 불릿 나열 금지, 제목은 완성된 동사 문장 |

피드백 메모리는 **사용자 정정** 또는 **칭찬** 직후에 저장된다. "그렇게 하지 마" 와 "그래 그게 맞아" 둘 다 학습 신호. 다음 세션은 같은 정정을 두 번 받지 않는다.

## 2. 커스텀 슬래시 커맨드

`.claude/commands/` 디렉토리의 마크다운 파일이 곧 슬래시 커맨드로 등록된다. 자주 쓰는 작업을 짧은 명령으로 압축한다.

| 커맨드 | 정의 | 동작 |
|---|---|---|
| `/test` | [`commands/test.md`](../commands/test.md) | `./gradlew test` 실행 후 실패만 요약 (스택 전체 X — 클래스 / 메서드 / 핵심 메시지만) |
| `/grade` | [`commands/grade.md`](../commands/grade.md) | `grader` 서브에이전트로 채점. 6개 영역 100점 만점 + 영역별 근거 + Top 3 개선 |
| `/review-pr` | [`commands/review-pr.md`](../commands/review-pr.md) | 현재 브랜치 디프를 3개 리뷰어 (correctness / adversarial / testing) 에 **단일 메시지 다중 tool_use 로 병렬 호출** 후 통합 보고 |

각 커맨드 정의는 다음 구조다:

```markdown
---
description: <한 줄 설명>
argument-hint: "[선택: 인자 설명]"
---

<프롬프트 본문 — Claude 가 이 메시지를 받은 것처럼 동작>
```

`/review-pr` 의 핵심은 *병렬 실행* 강제다. 정의 파일에 `반드시 단일 메시지 다중 tool_use 로 병렬 실행. 순차 호출 금지.` 라고 명시되어 있어, 세 리뷰어 응답이 거의 동시에 돌아온다. 순차 실행 시 한 리뷰어가 다른 리뷰어 결과를 보고 영향받는 문제가 생기는데, 병렬은 그걸 차단한다.

## 3. 커스텀 서브에이전트 — `grader`

`.claude/agents/grader.md` 가 핵심. 일반 모델에게 *"엄격하게 채점해라"* 만 부탁하면 톤이 매번 다르고 점수 분포가 일정하지 않다. **시스템 프롬프트로 채점 룰북을 못박은 별도 컨텍스트**가 필요하다.

룰북 골격:

```
역할 정의 (채점자, 근거 기반)
   ↓
평가 기준 6개 영역 (각 배점 + 핵심 점검)
   ↓
도메인 점검 항목 10개 (멱등성 / 재시도 / 다중 인스턴스 안전성 등)
   ↓
감점 시그널 (사용처 없는 추상화, AI slop 주석 등)
   ↓
출력 형식 (점수 표 + 영역별 근거 + Top 3)
   ↓
행동 원칙 (파일·줄 단위 인용, 보수적 분포)
```

이 룰북이 있으면 같은 코드를 두 번 채점해도 결과가 거의 일치한다. 톤이 흔들리지 않는다. PR 머지 직전 `/grade` 한 번 호출이 회고·검증·우선순위 갱신을 동시에 해 준다.

### 메인 컨텍스트 vs 서브에이전트

| 측면 | 메인 컨텍스트 | `grader` 서브에이전트 |
|---|---|---|
| 시스템 프롬프트 | Claude Code 기본 + CLAUDE.md | 채점 룰북 |
| 도구 권한 | 전체 | Read, Grep, Glob, Bash 만 (수정 불가) |
| 컨텍스트 격리 | 진행 중 작업 / 대화 누적 | 새 세션처럼 시작 — 이전 작업의 자기 변호 영향 없음 |
| 호출 시점 | 평소 | `/grade` 또는 사용자 요청 시 |

쓰기 권한이 없어 *"채점하다가 답을 고쳐서 점수를 올리는"* 경로가 막혀 있다.

## 4. compound-engineering 플러그인 — 외부 리뷰어 활용

`compound-engineering` 플러그인이 제공하는 리뷰어 에이전트들을 `/review-pr` 에서 활용한다.

| 리뷰어 | 초점 | 관점 |
|---|---|---|
| `ce-correctness-reviewer` | 로직 오류, 엣지 케이스, 상태 관리 버그, 의도-구현 불일치 | "이 코드가 명세대로 동작하는가" |
| `ce-adversarial-reviewer` | 깨뜨릴 수 있는 시나리오 능동 구성 (동시성, 부분 실패, 우회 경로) | "이 코드를 어떻게 깨뜨릴 수 있나" |
| `ce-testing-reviewer` | 커버리지 갭, 약한 단언, 구현에 결합된 부서지기 쉬운 테스트 | "이 테스트가 회귀를 잡을 수 있나" |

세 리뷰어는 **상보적이다** — 같은 디프를 봐도 잡는 이슈가 다르다. correctness 는 명세 기반 검증, adversarial 은 명세에 없는 공격 시나리오, testing 은 미래 회귀 방지. 한 리뷰어로는 사각지대가 생긴다.

병렬 실행이 끝나면 통합 단계에서:

- 같은 이슈를 두 리뷰어가 잡으면 **출처를 둘 다 표시** 해 한 항목으로 합침
- HIGH 항목은 머지 전 반드시 처리
- 통합 단계에서 **새 이슈 추가 금지** — 리뷰어 결과만 정리 (의견은 별도 "## 추가 메모" 섹션)

## 5. 워크트리 — 병렬 브랜치 작업

`compound-engineering:ce-worktree` 스킬을 활용해 브랜치마다 독립된 작업 디렉토리를 둔다.

```
notification-practice/
├── (메인 worktree, main 브랜치)
└── .claude/worktrees/
    ├── feature-1-xxx/   (브랜치 A)
    ├── feature-2-yyy/   (브랜치 B)
    └── ...
```

장점:

- 한 PR 작성 중에 다른 PR 의 리뷰 코멘트에 즉시 응답 가능 — 브랜치 전환에 따른 빌드 캐시 / IDE 인덱스 무효화가 없다
- 워크트리마다 독립 Claude 세션 — 컨텍스트가 섞이지 않음
- 작업이 끝난 워크트리는 삭제만 하면 정리 완료

단점:

- 디스크 사용량이 늘어남 (Gradle 캐시 등이 워크트리마다 별도)
- 병렬 작업 수가 많아지면 어떤 워크트리가 어떤 PR 인지 헷갈림 → 디렉토리명에 의도를 담는다

## 6. 작업 흐름 — 한 PR 의 라이프사이클

```
1. 로드맵에서 다음 PR 선택            (.claude/docs/roadmap.md)
       ↓
2. 워크트리 + 브랜치 생성             (compound-engineering:ce-worktree)
       ↓
3. 구현                              (사람: 설계 결정 / Claude: 코드 작성)
       ↓
4. 테스트 작성 + /test 실행           (슬라이스 분리, 실패 시나리오 포함)
       ↓
5. /review-pr (3 리뷰어 병렬)         (HIGH 항목은 PR 전 반드시 처리)
       ↓
6. /grade — 100점 채점                (점수 + 우선순위 Top 3)
       ↓
7. PR 작성 + 머지                    (Git 컨벤션 준수, AI Co-author 라벨)
       ↓
8. 워크트리 삭제                      (정리)
```

### 자기 강화 루프

6번 `/grade` 결과는 채점에서 끝나지 않는다 — 다음 PR 의 입력이 된다.

- *"문서화 8/10"* → 다음 PR 의 README 보강 항목 추가
- *"테스트 약한 단언"* → 다음 PR 의 테스트 슬라이스 보강
- *"미구현 명시 부족"* → README 의 "미구현 / 제약" 섹션 갱신

채점 결과 → 우선순위 갱신 → 다음 PR → 채점 결과의 사이클이 누적되면서 코드 / 문서가 동시에 다듬어진다.

### 사람 / Claude 분담 — 단계별

| 단계 | 사람 | Claude |
|---|---|---|
| 1. PR 선택 | ✅ 의사결정 | — |
| 2. 워크트리 생성 | — | ✅ 명령 실행 |
| 3. 구현 — 설계 결정 | ✅ 비교·선택 | — |
| 3. 구현 — 코드 작성 | 검토만 | ✅ 작성 |
| 4. 테스트 작성 | 시나리오 결정 | ✅ 작성 |
| 5. 셀프 리뷰 | HIGH 처리 결정 | ✅ 3 리뷰어 호출 |
| 6. 채점 | 결과 해석 | ✅ grader 호출 |
| 7. PR 작성 — 본문 | 톤 / 의도 검토 | ✅ 초안 작성 |
| 7. 머지 | ✅ 의사결정 | — |

## 7. Git 훅 — 자동 검증 게이트

[`.githooks/`](../../.githooks/) 디렉토리에 git 훅을 두고 `git config core.hooksPath .githooks` 로 연결한다. 별도 install 스크립트 없이 한 줄로 활성화:

```bash
git config core.hooksPath .githooks
```

| 훅 | 동작 | 목적 |
|---|---|---|
| `pre-commit` | `./gradlew compileKotlin compileTestKotlin` | 컴파일 깨진 채로 커밋되지 않게 |
| `pre-push` | `./gradlew ktlintCheck test` | 린트 / 테스트 깨진 채로 푸시되지 않게 |

훅은 Claude 가 `--no-verify` 로 우회하지 않게 하기 위한 추가 명시는 두지 않았다 — 훅이 실패하면 에이전트가 자연스럽게 실패 사유를 보고 고치고 재시도한다. 우회는 사용자가 명시적으로 허락할 때만 한다.

## 8. AI 활용 범위 — 무엇을 위임했고, 무엇을 위임하지 않았는가

### Claude 에 위임하지 않은 의사결정 (사람이 직접 한 것)

- **비동기 구조 선택** — outbox vs 인메모리 vs `@Async` 비교 후 outbox 결정
- **멱등성 키 생성 주체** (서버) + 키 구성 인자 결정 (`scheduledAt` 포함 여부)
- **분산 락 방식** — `FOR UPDATE SKIP LOCKED` vs ShedLock 비교 후 전자 결정
- **DEAD_LETTER 카운터 분리 정책** (자동 / 수동 독립)
- **PR 단위 분리 + 로드맵 작성** (PR 1 ~ PR 9 의 의존 관계와 단위)
- **"메시지 브로커 없이"의 해석** (보수적 — Redis Streams 도 회색지대로 판단해 제외)
- **처리 타임아웃 임계값 결정** (5 분)
- **README / 문서의 톤과 구성**

### Claude 에 위임한 작업

- 위 의사결정에 따른 구현 코드 작성
- 슬라이스 테스트 작성 (`@DataJpaTest`, `@WebMvcTest`, `@SpringBootTest`)
- 셀프 리뷰 (3 리뷰어 병렬) + 채점
- ktlint / 컴파일 에러 수정
- 문서 초안 작성 (이후 사람이 톤 / 구성 검토)
- PR 본문 / 커밋 메시지 초안

### 사용한 Claude 도구 정리

| 도구 | 용도 |
|---|---|
| Claude Code | 코드 / 테스트 / 문서 작성, 셀프 리뷰, 채점 |
| 커스텀 서브에이전트 (`grader`) | 평가 룰북 기반 채점 (자체 검증) |
| `compound-engineering` 리뷰어 3종 | PR 전 셀프 리뷰 (correctness / adversarial / testing) |
| `compound-engineering:ce-worktree` | 워크트리 관리 |
| 사용자 메모리 (`MEMORY.md`) | 세션 간 피드백 학습 유지 |
| Git 훅 | 컴파일 / 린트 / 테스트 자동 검증 게이트 |

## 9. 한계와 의식적으로 두지 않은 것

- **자동 PR 머지를 두지 않았다** — 모든 PR 은 사람의 머지 클릭이 필요하다. 채점 결과가 좋아도 자동 머지하지 않는 게 의식적인 선택이다 (압도적으로 좋은 자동화 < 모르는 사이 새는 코드).
- **CodeRabbit 같은 자동 리뷰 도구를 거부한다** — 외부 리뷰어가 PR 마다 수십 개 코멘트를 남기는 패턴은 신호 대 노이즈가 낮다. 사용자 메모리(`feedback_no_coderabbit.md`) 에 명시.
- **AI 가 만든 코드라는 표시는 PR 본문에 한 줄 명시까지만** — 라인별 주석에 `// generated by AI` 같은 메타데이터는 두지 않는다. 코드는 사람이 책임지고 머지한 코드일 뿐이다.
- **Claude 의 출력 그대로 머지하지 않는다** — 모든 변경은 사람이 리뷰 후 머지한다. AI 는 정리·초안·검토의 가속 도구로 사용했다. 설계 결정과 코드 책임은 사람에게 있다.
