# Git 컨벤션

## 브랜치

```
<type>/<scope>-<short-description>
```

| 타입 | 사용 |
|---|---|
| `feat` | 기능 추가 |
| `fix` | 버그 수정 |
| `refactor` | 동작 변경 없는 구조 개선 |
| `chore` | 빌드/도구/CI/의존성 |
| `docs` | 문서 |

`<scope>` 는 도메인 영역 (예: `notification`, `worker`, `idempotency`).

예시: `feat/notification-idempotency-key`, `refactor/worker-polling`

---

## 커밋

### 형식

```
<type>(<scope>): <한줄 요약>

<본문 — 의도 / 결정 근거 / 영향 범위>
```

### 타입

| 타입 | 사용 |
|---|---|
| `feat` | 새 기능 |
| `fix` | 버그 수정 |
| `refactor` | 동작 변경 없는 구조 개선 |
| `test` | 테스트 추가/수정 |
| `docs` | 문서 |
| `chore` | 빌드/도구/CI/의존성 |

### 원칙

- 한 커밋 = 한 의미. 두 가지 변경은 두 커밋으로 분리한다.
- 리팩토링은 별도 커밋. `feat` 안에 정리 작업을 섞지 않는다.
- 본문에는 "왜" 를 적는다. "무엇" 은 diff 가 보여준다.
- 제목은 명령형, 마침표 없이 작성한다.

### 예시

```
feat(notification): 멱등성 키와 DB UNIQUE 제약 도입

같은 이벤트가 재발행되어도 row 1건만 보장하기 위해 SHA-256 기반
키와 UNIQUE 제약을 도입한다. duplicate key 예외는 무시하고 정상
응답으로 처리해 발행자 관점에서 멱등하게 동작하도록 한다.
```

## PR

### 제목

커밋과 동일한 형식. **한국어로 작성**.

```
<type>(<scope>): <한줄 요약>
```

예시:

- `feat(notification): 멱등성 키와 DB UNIQUE 제약 도입`
- `refactor(worker): 폴링 루프를 별도 트랜잭션으로 분리`
- `chore(infra): 앱·DB 포트 비기본값으로 변경`

### 본문 구성

고정 템플릿이 아니다. **Summary 와 Test plan 만 항상 포함**하고, 가운데는 변경 성격에 맞춰 필요한 섹션만 사용한다.

```markdown
## Summary

<항상 포함. 1~3문장. 시스템·사용자 관점의 변화와 의도.>

<— 변경 성격에 맞는 섹션을 골라서 작성. 아래는 자주 쓰는 후보. —>

## Test plan

<항상 포함. 자동 검증 + 수동 검증 체크박스.>
```

### 자주 쓰는 본문 섹션 후보

| 섹션 | 언제 쓰는가 |
|---|---|
| `### 문제` / `### 배경` | 발견 경위, 재현 조건, 영향 범위 설명이 필요할 때 (버그·인시던트 대응 PR) |
| `### 수정` / `### 변경 내역` | 변경 항목을 나열해야 할 때 (대부분의 PR) |
| `### 접근` / `### 설계` | 채택한 방식과 거른 대안 비교가 의사결정에 중요할 때 (큰 리팩토링·아키텍처 변경) |
| `### 영향` / `### 호환성` | 기존 동작·API·운영 시나리오에 변화가 있을 때 |
| `### 마이그레이션` / `### 머지 전 수동 작업` | 환경변수 추가, DB 마이그레이션, 외부 설정 등 자동화되지 않은 단계가 있을 때 |
| `### 롤백` | 롤백 절차가 단순하지 않을 때 |
| `### 스크린샷` / `### 데모` | UI/응답 형식이 달라졌을 때 |

조합 예:

- 작은 버그 수정 → `Summary` / `문제` / `수정` / `Test plan`
- 인프라 변경 → `Summary` / `변경 내역` / `머지 전 수동 작업` / `롤백` / `Test plan`
- 큰 리팩토링 → `Summary` / `배경` / `접근` / `변경 내역` / `영향` / `Test plan`
- 문서만 → `Summary` / `Test plan` (체크박스 1~2개만으로 충분)

### 원칙

- Summary 는 변경 파일 나열이 아니라 **시스템·사용자 관점의 변화**를 적는다.
- 본문 섹션은 변경 성격에 맞게 추가·생략한다. 빈 섹션을 형식 맞추기로 남기지 않는다.
- 머지 전 수동 작업이 있으면 반드시 명시한다 (운영 환경변수, DB 마이그레이션 등).
- Test plan 의 모든 체크박스는 머지 시점에 ✅ 되어 있어야 한다. 자동 검증이 어려운 항목은 수동 검증 절차로 명시한다.
