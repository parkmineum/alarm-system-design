# 요구사항 해석과 개선 의견

이 문서는 과제 요구사항을 어떻게 해석했는지, 어떤 가정을 추가했는지, 어떤 항목을 의도적으로 제외했는지, 그리고 실제 운영 시스템으로 발전시킨다면 우선적으로 손볼 지점을 정리한다.

---

## 1. 요구사항 한 줄 정리

> 다양한 비즈니스 이벤트(수강 신청, 결제 확정, 강의 D-1, 취소 등) 가 발생할 때, **비즈니스 트랜잭션을 깨지 않으면서**, **중복 없이**, **유실 없이** 사용자에게 EMAIL / IN_APP 알림을 발송하는 시스템.

---

## 2. 모호한 지점과 우리의 해석

### 2.1 "메시지 브로커 없이" 의 경계

> 외부 의존성 제약: 실제 메시지 브로커 설치 불필요. 단, 운영 환경 전환 가능한 구조 유지.

가장 해석이 갈리는 지점이다. 다음 단계로 분류해 결정했다.

| 카테고리 | 도구 예시 | 도입 여부 |
|---|---|---|
| 명시적 메시지 브로커 | Kafka, RabbitMQ, AWS SQS | ❌ |
| 메시지 큐 역할 가능한 인메모리 인프라 | Redis Streams / Pub-Sub | ❌ (회색지대지만 보수적으로 제외) |
| 일반 인프라 (영속 저장소 / 캐시) | MySQL, PostgreSQL | ✅ |
| 애플리케이션 레벨 도구 | Spring `@Async`, `@Scheduled` | ✅ |

Redis 는 분산 락(Redisson), 큐(Streams), 캐시 등 다양한 용도로 쓰일 수 있어 "브로커냐 아니냐" 가 모호하다. 보수적으로 도입하지 않고, 동등한 기능을 DB 로 대체했다.

| 본래 Redis 용도 | DB 대체 |
|---|---|
| 분산 락 | `SELECT ... FOR UPDATE SKIP LOCKED` |
| 큐 | Outbox 폴링 |
| 멱등성 키 캐시 | DB UNIQUE 제약 |
| Rate limiting | (필요 시 별도 도입) |

### 2.2 멱등성 키 생성 주체

> 동시에 같은 요청이 여러 번 들어오는 경우도 고려한다.

요구사항은 "중복이 없어야 한다" 로만 명시되어 있고, 키를 누가 만드는지는 열려 있다. 두 옵션 중 **서버 생성**을 채택했다.

| 옵션 | 장점 | 단점 |
|---|---|---|
| 클라이언트(호출자) 제공 | 호출자 의도를 그대로 키에 반영 | 호출자마다 키 정책이 다르면 일관성 깨짐 |
| **서버 생성 (채택)** | 한 곳에서 일관된 키 정책 유지 | 호출자가 "굳이 다른 발송" 을 원할 때 표현 어려움 |

`(eventType, refType, refId, recipientId, channel)` 다섯 축으로 SHA-256 해시한다. 다섯 축이 같으면 같은 발송으로 본다. 같은 사용자에게 같은 강의의 D-1 알림을 두 번 만들 일은 비즈니스적으로 거의 없으므로 이 키 정책으로 실질적 충돌은 발생하지 않는다.

### 2.3 "처리 중 상태가 일정 시간 이상 지속되는 경우"

> 처리 중 상태가 일정 시간 이상 지속되는 경우 복구 방법 설계

"일정 시간" 의 임계값이 명시되지 않았다. **10 분** 으로 가정했다. 외부 게이트웨이의 일반적 타임아웃(30~60s) 보다 충분히 길어 정상 발송이 잘못 복구되는 일이 적고, 사용자 체감 지연도 크지 않다.

운영 단계에서는 채널별로 분리할 가치가 있다. 인앱은 1분이면 충분, 이메일은 10분, 외부 푸시는 더 짧은 식.

### 2.4 "다중 인스턴스 환경"

> 다중 인스턴스 환경에서도 동일 알림이 중복 처리되지 않아야 한다.

"인스턴스" 는 같은 애플리케이션의 수평 확장으로 해석했다. 즉 워커 N 개가 동시에 같은 DB 를 보는 상황. 멀티 리전이나 활성-활성 DB 시나리오는 범위 밖.

### 2.5 "발송 실패" 의 정의

> 발송 실패 시 재시도 정책과 최종 실패 처리 방식 설계

외부 게이트웨이의 응답을 기준으로 한다. "성공" 으로 응답했지만 사용자가 메일을 못 받는 경우는 시스템 단독으로 감지 불가능하므로, 이 시스템에서는 게이트웨이 응답 = 진실로 가정한다. (실제 운영에서는 webhook / bounce notification 으로 보강 필요 — §4 참조)

## 3. 명시적이지 않지만 추가한 가정

| 가정 | 이유 |
|---|---|
| 알림 순서 보장은 하지 않는다 | 워커 동시성·재시도로 자연스럽게 어긋남. 비즈니스 레이어가 처리할 영역. |
| `Notification` row 는 영구 보관 (TTL 없음) | 운영 감사·CS 대응 / 통계 용도. 보관 정책은 실제 운영 시 별도로 결정. |
| 채널은 `EMAIL` / `IN_APP` 둘 뿐 | 과제 명시. PUSH, SMS, KakaoTalk 등은 어댑터 추가만으로 확장 가능. |
| 한 번의 등록 요청은 한 채널만 발송 | 한 이벤트를 두 채널로 보내려면 호출 측이 두 번 요청한다. fanout 은 §4. |
| 수신자 본인 인증은 시스템 외부에서 끝났다 | 과제 제약(인증 간략화). `userId` 가 헤더로 들어옴. |
| 메일 수신자 주소·인앱 디바이스 토큰 등은 별도 사용자 서비스에서 가져온다 | 알림 시스템은 발송 책임만. 발송 시점에 lookup. |
| 시간은 `Instant` (UTC) 보관, 표시 시점에만 KST 변환 | 다중 리전 / 협업 환경에서 표준. CLAUDE.md 컨벤션과 일치. |

## 4. 의도적으로 제외한 것

다음은 유의미하지만 5일 안에 모두 처리하기엔 깊이가 부족해 의도적으로 범위 밖으로 두었다.

| 항목 | 제외 이유 |
|---|---|
| 실제 메일 발송 (SES / SendGrid 연동) | 과제 명시 (Mock/로그 대체) |
| 사용자별 채널 선호도 / opt-out | 도메인 영역. 사용자 서비스의 책임. |
| 발송 결과 webhook / bounce 처리 | 외부 게이트웨이 의존, Mock 환경에서는 시뮬레이션이 무의미 |
| 분당 / 채널당 rate limit | 외부 게이트웨이마다 다름, 실제 게이트웨이 도입 시점에 결정 |
| 다국어 / 지역화 | 템플릿 모델은 확장 가능하지만, 실제 i18n 자원은 사용자 서비스 영역 |
| 알림 우선순위 (예: D-1 vs 결제 확정) | 큐 분리 / 가중치 폴링 등으로 가능하지만 과제 범위에서는 불필요 |
| Schema 마이그레이션 도구 (Flyway / Liquibase) | 단일 PR 단위 개발. 실제 도입 시 1~2일 작업. |
| 배포 / 헬스 체크 / 롤링 업데이트 시나리오 | 과제 범위 밖 |

## 5. 개선 의견 — 실제 운영 시 우선 손볼 지점

채점 기준 안에서는 모든 필수 요건이 충족되지만, 실제 서비스로 발전한다면 다음 순서로 보완하는 편이 가성비가 좋다.

### 5.1 즉시발송 경로 추가 (low-hanging fruit)

폴링 주기(500ms) 만큼 발송 지연이 누적된다. `AFTER_COMMIT` 훅에서 워커 풀에 즉시 enqueue 하고, 폴링은 안전망으로 두는 하이브리드. 정상 경로 지연이 거의 0 이 된다.

- 변경 범위: dispatcher 한 곳
- 위험: 즉시발송 실패 시에도 폴링이 다음 사이클에 픽업하므로 안전
- 비용: 코드 ~30 줄

### 5.2 DEAD_LETTER 운영 대시보드

DLQ 가 쌓이는데 운영자가 모르는 게 가장 큰 운영 리스크다. 다음 정도가 있으면 충분하다.

- 채널별 / 사유별 카운트
- 최근 1시간 / 24시간 증가 추이
- 사유 텍스트 클러스터링 (`last_error` 의 prefix 기준)
- 일괄 재시도 / 일괄 폐기 액션 (감사 로그 동반)

### 5.3 사용자 단위 throttle / digest

같은 사용자에게 짧은 시간 내 다수 알림이 트리거되면 전부 발송된다. 결제 N 건이 동시에 일어난 사용자가 메일 N 통을 받는 식. 디지스트(시간 단위로 묶어서 1통) 또는 throttle(N분에 1통) 정책이 필요하다.

- 모델: `(recipient_id, channel, time_bucket)` 단위 카운터
- 정책 적용 위치: 발송 직전 워커에서 체크 (요청 시점이 아님 — 비동기 누적이 핵심)

### 5.4 채널 fanout

같은 이벤트를 여러 채널로 동시 발송하는 일이 흔하다 ("결제 완료" 를 메일 + 인앱 + 푸시). 현재는 호출자가 채널마다 한 번씩 호출하는 구조. fanout 계층을 두면 호출자가 한 번만 알리고 알림 시스템이 사용자 선호도에 따라 채널을 선택한다.

```
호출자 → Notification (이벤트 1 개) → fanout → NotificationChannel (채널별 N 개)
```

`Notification` 과 `NotificationChannel` 을 분리하고, 사용자 선호도를 lookup 해 채널 N 개로 분기.

### 5.5 외부 게이트웨이 추상화 강화

현재 `NotificationSender` 는 단순 인터페이스. 실제 SES/SendGrid 도입 시:

- API 키 / 엔드포인트 / 리전 설정 분리
- 게이트웨이별 rate limit 설정
- 부분 실패 (배치 호출 시 일부만 실패) 처리
- bounce / complaint webhook 수신

### 5.6 메트릭과 알람

운영 단계로 가면 SQL 만으로는 부족하다. Micrometer + Prometheus 조합 권장:

| 메트릭 | 타입 | 의미 |
|---|---|---|
| `notification.dispatch.count` | Counter (channel, status) | 발송 결과 추적 |
| `notification.queue.depth` | Gauge | PENDING 누적 |
| `notification.processing.duration` | Timer (channel) | 외부 호출 지연 |
| `notification.retry.count` | Counter (attempt) | 재시도 빈도 |
| `notification.deadletter.count` | Counter (channel, reason) | DLQ 증가 |

### 5.7 Schema 마이그레이션 도구 도입

JPA `ddl-auto = update` 는 개발/데모용. 운영에서는 Flyway 또는 Liquibase 로 마이그레이션을 코드 리뷰 가능한 SQL 로 관리해야 한다. 동시에 컬럼 추가/삭제 절차(blue-green DDL) 도 정해야 한다.

### 5.8 알림 보관 정책

`Notification` row 가 영구히 쌓이면 테이블이 커진다. 다음 정책이 흔하다.

- 90일 이후 cold storage 이관
- 1년 이후 hash-only 보존 (감사용, 본문 폐기)
- 사용자 탈퇴 시 즉시 삭제 (개인정보보호)

## 6. 구현하지 않은 채점 항목과 그 이유

| 항목 | 상태 | 사유 |
|---|---|---|
| 실제 메일 발송 | Mock | 과제 명시 |
| Kafka 등 브로커 연동 | 추상 인터페이스만 | 과제 제약 |
| 운영 대시보드 UI | 미구현 | 과제 범위 밖, §5.2 에서 제안 |
| 사용자 선호도 / opt-out | 미구현 | 사용자 서비스 영역 |
| 다국어 템플릿 자원 | 단일 언어 | i18n 자원 시스템이 별도 필요 |
