# 요구사항 해석 및 개선 의견

## 해석 및 가정

### 멱등성 키 생성 주체

과제는 "동일 이벤트에 대해 중복 발송 방지"를 요구하지만 멱등성 키 포맷을 명시하지 않았다. 호출자에게 키를 직접 넘기도록 할 수도 있지만, 호출자가 키를 잘못 생성하는 오류 가능성을 줄이기 위해 서버가 `(type, refType, refId, recipientId, channel, scheduledAt?)` 조합으로 SHA-256을 계산하도록 결정했다.

이 방식의 제약: 같은 이벤트를 다른 채널로 보내는 것은 별도 알림으로 취급된다(채널이 키에 포함됨).

### 발송 채널

`EMAIL`과 `IN_APP` 두 채널만 구현했다. 실제 발송은 Mock(로그 출력)으로 대체했으며, `NotificationSender` 인터페이스를 구현하면 실제 외부 연동이 가능하다.

### 인증/인가

`X-User-Id` 헤더를 신뢰하여 수신자 본인 확인에 사용한다. 토큰 기반 인증은 미구현이며, 실제 서비스에서는 API Gateway 또는 Spring Security에서 검증된 사용자 ID를 주입하는 방식으로 교체 가능하다.

### 스케줄링 기준

`scheduledAt`을 생략하면 등록 시각이 기본값이 된다. 워커는 `scheduled_at <= NOW()`인 PENDING만 폴링하므로, 과거 시각이면 즉시 발송 대상, 미래 시각이면 해당 시각이 되기 전까지 대기한다.

## 개선 의견

### 폴링 대신 이벤트 드리븐

현재 구조는 1초 주기 폴링이다. DB 부하를 줄이고 발송 지연을 최소화하려면 Kafka나 RabbitMQ 같은 메시지 브로커를 도입할 수 있다. `OutboxDispatcher` → `NotificationDispatcher` 인터페이스 구조가 이 전환을 위해 설계되었다. 등록 시 이벤트를 발행하고 컨슈머가 처리하는 방식으로 전환하면 폴링 지연이 사라진다.

### 수신함 목록 페이지네이션

`GET /users/{userId}/notifications`는 현재 전체 목록을 반환한다. 알림이 누적되면 메모리 부담이 생기므로 Pageable을 적용하는 것이 바람직하다. DLQ 목록은 이미 페이지네이션이 적용되어 있다.

### DEAD_LETTER 알림 만료 정책

현재 DEAD_LETTER는 영구 보관된다. 일정 기간(예: 30일) 이후 아카이브하거나 삭제하는 정책이 없으면 테이블이 계속 커진다. 만료 배치 잡 또는 파티셔닝을 고려할 수 있다.

### 관리자 인증

`X-Actor-Id` 헤더는 단순 문자열이다. 실제 운영에서는 관리자 권한 검증이 필요하다. 현재 구조에서는 누구나 헤더 값만 알면 수동 재시도 API를 호출할 수 있다.

### 알림 템플릿

현재 `payload` 필드에 JSON 원문을 저장한다. 알림 타입별 메시지 포맷이 고정되어 있다면 `NotificationTemplate` 엔티티를 도입하고, 워커가 발송 직전에 렌더링하는 구조로 발전시킬 수 있다. 템플릿을 바꿔도 재배포 없이 메시지 포맷을 변경할 수 있다는 장점이 생긴다.
